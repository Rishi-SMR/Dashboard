01

Purpose & scope

The SMR Cashflow Dashboard is a read-and-post financial console over Sports Med Recovery's Striven ERP. It answers four questions the ERP makes slow to answer: what are we owed, what do we owe, what is moving, and what is stuck — and it posts approved invoices into QuickBooks Online without re-keying.

ItemDetail
Who uses itPractice owner, finance/admin staff, and the external accountant. Three named accounts today.
System of recordStriven ERP. The dashboard never becomes the source of truth — it reads, shapes and posts.
RefreshCached in Supabase, refreshed automatically every 6 hours; each screen also has a manual refresh.
Out of scopeClinical notes, diagnoses, imaging, insurance IDs, dates of birth, addresses. The dashboard must not be extended to carry these without a fresh risk analysis.

The one design rule behind everything

No patient name ever leaves Striven. Every screen, every export and every record posted to QuickBooks refers to a patient as PT-418 — a reference built from their Striven customer id. Staff who need the actual name look it up in Striven, where access is already controlled.

Rules that are enforced by the system, not just written down

ControlBehaviour
Password storagescrypt-hashed. Even with database access, passwords cannot be read.
Failed attempts5 wrong passwords in 15 minutes locks that account for 15 minutes.
IdentityEach session token is signed and names one user. There is no shared login.
AuditEvery login and every read of patient-derived data is recorded with user, page, IP and time.
Failure modeIf the user directory is unreachable the gate refuses everyone rather than falling back to a weaker credential.

Adding or removing a user

  1. The Security Officer approves in writing, naming the person and the business reason.
  2. The maintainer adds a row to dashboard_users with a hashed password — never plaintext.
  3. The password is delivered out-of-band; the user completes training before first login.
  4. On departure: delete the row the same day. To kill a live session immediately, rotate SESSION_SECRET — this logs everyone out and is the correct move for any non-amicable exit.

04

Procedure: posting to QuickBooks

Run in order. Each step only shows what is genuinely missing, so a step with nothing listed is already reconciled and can be skipped.

StepActionWhy the order matters
1 · CustomersReview the list of Striven customers missing from QuickBooks. Create all, or tick a selection.An invoice cannot post without its customer. Created as PT- references.
2 · VendorsSame, for suppliers.Needed before any bill or purchase-side record.
3 · ItemsSame, for products and services.Invoice lines reference items; missing items fail the post.
4 · InvoicesFilter Pending / Posted / All. Preview an invoice, confirm, then post — singly, as a selection, or all pending.Final step, once the three above are clean.

Two behaviours worth knowing

  • Dates are preserved. A posted invoice carries its original Striven date, not the date you clicked post — so the accounts stay in the correct period.
  • Nothing posts twice. Posted invoices are tracked, so re-running a batch skips what is already in QuickBooks.

Never do this

Do not send a patient name, date of birth, address, phone, email, diagnosis or any clinical description to QuickBooks — no matter how convenient a future feature makes it. Intuit is not covered by a BAA, so anything sent there is an unprotected disclosure.

An important distinction: PT-418 is not "anonymous"

Replacing names with references is pseudonymisation, not de-identification. Because SMR keeps the key in Striven, the order history and amounts stored against PT-418 remain protected health information under §164.514(b). This matters practically: it means Supabase and Vercel are still Business Associates and still need BAAs. What the references genuinely achieve is a much smaller blast radius if anything leaks, and keeping Intuit out of the PHI chain entirely.

Procedure: verifying compliance monthly

Step 1 — run the automated check

$ node scripts/hipaa-check.mjs

PASS  A1  Passwords are hashed, never plaintext
PASS  A3  Unauthenticated requests are refused
PASS  A4  A forged session token is rejected
PASS  A6  Brute force is locked out
PASS  A7  Session cookies are HttpOnly + Secure
PASS  B2  Every read of patient data is attributed to a user
PASS  C1  No patient name is stored in the Supabase cache
PASS  C3  QuickBooks holds no patient name
PASS  D1  Site is served over TLS only

Technical controls: 13/13 PASS

Safe to run any time — read-only, and the lockout test uses a throwaway username so it can never lock a colleague out. A failure prints the exact JSON path of the problem. Do not silence a failure by adding it to the allowlist — justify it in writing first.

Step 2 — review the audit logs (a human must do this)

-- failed logins in the last 30 days
select username, ip, at from login_events
where success = false and at > now() - interval '30 days' order by at desc;

-- who read patient data, and how much
select username, count(*), min(at), max(at) from phi_access_events
where at > now() - interval '30 days' group by username order by 2 desc;

Look for logins by departed staff, unfamiliar locations, bursts of failures, or a volume far outside someone's normal pattern. Record that the review happened even when nothing was found — the record is the deliverable, not the reassurance.

Step 3 — tick the manual items

BAAs signed · training register current · user list matches actual staff · risk analysis reviewed within 12 months · Security Officer named. A script cannot check any of these, and they are what an auditor asks for first.

Also re-run the check before shipping any change that…

  • touches login, sessions or cookies
  • adds a new cached dataset, or a new field to an existing one
  • changes anything posted to QuickBooks
  • adds any new external service — which needs its BAA signed first

Daily rules for everyone with a login

DoDo not
Log in only from an encrypted work device with a screen lock Log in from a public, shared or hotel computer
Use a unique password of 12+ characters Reuse a password from email or banking
Refer to patients by their PT- reference in email and chat Share your login — every action is recorded against your name
Delete CSV exports as soon as the task is finished Email or upload an export outside SMR
Report anything odd the same day Paste dashboard data into ChatGPT, Claude or any unapproved online tool
Reach the site from your bookmark Look up a patient, friend or colleague out of curiosity — every read is logged

If something goes wrong

Tell the Security Officer the same day — a lost device, a shared password, an export sent somewhere it should not have gone, a name appearing where a reference should be, or a login you do not recognise. Do not wait until you are sure it was real; fast reporting is what turns a breach into a near-miss. There is no penalty for reporting in good faith.

Summary in one paragraph

The software side is done and, unusually, it is verifiable — one command re-proves all 13 technical controls, so drift gets caught instead of assumed away. The remaining gap is organisational: three signatures, one appointment and one training session. None of it is engineering work, and all of it is achievable in a week. Until then the honest position is that the dashboard is built to HIPAA's technical standard but SMR is not yet HIPAA compliant.
