01

Purpose & scope

The Cashflow Dashboard is a financial console over Sports Med Recovery's Striven ERP. It answers four questions the ERP makes slow to answer — what are we owed, what do we owe, what is moving, and what is stuck — and posts approved invoices into QuickBooks Online without re-keying.

ItemDetail
Who uses itPractice owner, finance/admin staff, and the external accountant. Three named accounts today.
System of recordStriven ERP. The dashboard reads, shapes and posts — it never becomes the source of truth.
RefreshCached in Supabase, refreshed automatically every 6 hours; each screen also has a manual refresh.
Out of scopeClinical notes, diagnoses, imaging, insurance IDs, dates of birth, addresses.

The design rule behind everything

No patient name ever leaves Striven. Every screen, every export and every record posted to QuickBooks refers to a patient as PT-418 — a reference built from their Striven customer id. Staff who need the actual name look it up in Striven, where access is already controlled. This single decision is what keeps the dashboard, the database and the accounting system free of patient identifiers.

Adding or removing a user

  1. The Security Officer approves in writing, naming the person and the business reason.
  2. The maintainer adds a row to dashboard_users with a hashed password.
  3. The password is delivered out-of-band; the user completes training before first login.
  4. On departure: delete the row the same day. To end a live session immediately, rotate SESSION_SECRET — this logs everyone out and is the correct move for any non-amicable exit.

04

Procedure: posting to QuickBooks

Run in order. Each step only shows what is genuinely missing, so a step with nothing listed is already reconciled and can be skipped.

StepActionWhy the order matters
1 · CustomersReview the Striven customers missing from QuickBooks. Create all, or tick a selection.An invoice cannot post without its customer. Created as PT- references.
2 · VendorsSame, for suppliers.Needed before any bill or purchase-side record.
3 · ItemsSame, for products and services.Invoice lines reference items; missing items fail the post.
4 · InvoicesFilter Pending / Posted / All. Preview an invoice, confirm, then post — singly, as a selection, or all pending.Final step, once the three above are clean.

Two behaviours worth knowing

  • Dates are preserved. A posted invoice carries its original Striven date, not the date you clicked post — so the accounts stay in the correct period.
  • Nothing posts twice. Posted invoices are tracked, so re-running a batch skips what is already in QuickBooks.

The one rule for this screen

Never send a patient name, date of birth, address, phone, email, diagnosis or clinical description to QuickBooks — no matter how convenient a future feature makes it. Intuit does not offer a Business Associate Agreement for QuickBooks Online, which is precisely why the integration was designed to send references and amounts only.

HIPAA safeguards in place

Each of the following is implemented, running in production, and re-checkable on demand by the procedure in section 6.

Individual identityEvery user has their own account and a signed session token naming them. No shared login or shared password exists anywhere in the system.§164.312(a)(2)(i)
Passwords unreadableStored as scrypt hashes with per-user random salts. No plaintext copy exists in the code, configuration, environment or secret vault.§164.312(a)(2)(iv)
Full audit trailEvery login attempt and every read of patient-derived data is recorded with the user, the page, the IP address and the timestamp.§164.312(b)
Minimum necessaryPatient names are stripped at every write. Only references reach the browser, the database and the accounting system.§164.502(b)
Encrypted in transitTLS enforced site-wide; plain HTTP is redirected. Session cookies are HttpOnly and Secure, so they cannot be read by scripts or sent unencrypted.§164.312(e)(1)
Encrypted at restThe Supabase database encrypts stored data with AES-256.§164.312(a)(2)(iv)
Log-in monitoringFive failed attempts within fifteen minutes locks the account, blocking password-guessing attacks.§164.308(a)(5)(ii)(C)
Fails closedIf the user directory cannot be reached, every login is refused rather than falling back to a weaker credential. Access is never accidentally left open.§164.312(a)(1)
Credentials isolatedAll secrets are read server-side only. The browser never receives an API key, a database key or a credential of any kind.§164.312(a)(1)
Nothing identifying at IntuitQuickBooks holds references and amounts only — verified across all 8 customer records.§164.502(e)
Written policiesEight documents covering policies, risk analysis, incident response, retention, training and vendor agreements.§164.316(a)
Documented risk analysisTwenty-one risks identified and rated, each with an owner, a control and a target date.§164.308(a)(1)(ii)(A)

Why the reference design matters — and what it does not do

Replacing names with references means a breach of the dashboard or the database would expose order amounts against codes, not identified patients. It also keeps QuickBooks entirely outside the protected-data chain. That is a genuine and substantial reduction in exposure.

It is worth stating clearly, because it is often misunderstood: this is pseudonymisation, not de-identification. SMR keeps the key in Striven, so the underlying data remains protected health information and the vendors that host it remain Business Associates. That is why the agreements in section 8 still matter.

Run monthly, and before shipping any change to login, cached data or the QuickBooks integration.

PASS  A4  A forged session token is rejected
PASS  A6  Brute force is locked out

Technical controls: 13/13 PASS

Safe to run at any time — it is read-only, and the lockout test uses a throwaway username so it can never lock a colleague out. A failure prints the exact location of the problem.

Step 2 — review the audit logs

-- failed logins in the last 30 days
select username, ip, at from login_events
where success = false and at > now() - interval '30 days' order by at desc;

-- who read patient data, and how much
select username, count(*), min(at), max(at) from phi_access_events
where at > now() - interval '30 days' group by username order by 2 desc;

Look for logins by departed staff, unfamiliar locations, bursts of failures, or a volume far outside someone's normal pattern. Record that the review happened even when nothing was found — the record is what an auditor asks for.

Step 3 — confirm the standing items

Vendor agreements current · training register up to date · user list matches actual staff · risk analysis reviewed within twelve months.

If something goes wrong

Tell the Security Officer the same day — a lost device, a shared password, an export sent somewhere it should not have gone, a name appearing where a reference should be, or a login you do not recognise. Do not wait until you are sure it was real; fast reporting is what turns a breach into a near-miss, and there is no penalty for reporting in good faith.
